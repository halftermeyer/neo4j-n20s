noRDF: What Happens When You Stop Mapping RDF to LPG and Start Using Both

A Hybrid Architecture for Knowledge Graphs on Neo4j
Pierre Halftermeyer · Neo4j
June 2026
Abstract

For years, the RDF-to-LPG conversation has been about mapping: how do you translate triples into nodes and edges? But mapping is lossy. You flatten class hierarchies, lose inference semantics, and abandon provenance. This article explores a different question: what if you keep both?

noRDF ("not only RDF") is an architectural pattern that stores RDF natively inside Neo4j as reified triples, runs RDFS inference with full justification tracking, and then selectively projects the results into a labeled property graph—driven by a single ontology. The LPG handles what graphs do best: path traversals, centrality, community detection. The RDF layer handles what logic does best: type inference, class hierarchies, provenance chains. One database. Two paradigms. Zero mapping.

We present the design principles, the lpg: ontology that bridges both layers, scale results at 50,000 nodes, and a supply-chain BOM demo where Cypher scopes the problem and RDF reasoning detects regulatory violations.

§1. The Mapping Trap

Neo4j has had an excellent RDF story for years, thanks to n10s (neosemantics). You can import RDF, map classes to labels, properties to node attributes, and run Cypher on the result. It works. Thousands of teams use it in production.

But mapping comes with trade-offs. When you map rdfs:subClassOf to a Neo4j relationship, the label you see in the database is a Neo4j label—not an RDF resource you can reason over. When you map an owl:TransitiveProperty, you either pre-compute the full closure (expensive) or lose transitivity (incomplete). When you map provenance triples, they become properties on nodes—invisible to RDFS rules that produced them.

The mapping model answers one question: how do I see my RDF data in Neo4j? noRDF asks a different one: what if Neo4j could reason like an RDF engine AND traverse like a graph database, at the same time?

A note on n10s

n10s is not the problem—it's a mature, battle-tested tool with a clear use case: making RDF data accessible to Cypher users. noRDF starts from a different premise: that some workloads need both paradigms simultaneously, with RDF semantics kept alive for reasoning. The two approaches are complementary. If your RDF data is static and you need fast LPG queries, n10s is the right choice. If you need live inference, provenance tracking, and graph algorithms on the same dataset, read on.

§2. The Three Questions

Every knowledge system answers one of three questions. Most pick one. noRDF answers all three, each with the tool best suited to it.

QuestionParadigmToolWorld Assumption
"What else must be true?"InferenceRDFS/OWL rules via TMSOpen world, monotonic
"What can this database contain?"EnforcementGQL GRAPH TYPEClosed world, write-time
"What can be computed on demand?"ComputationCypher on LPG projectionStructural, query-time

2.1 Inference: What Else Must Be True?

If Retinol is a RetinoidAgent, and RetinoidAgent is a subclass of ActiveIngredient, then Retinol is an ActiveIngredient. You don't assert this—you infer it. RDFS handles this naturally via rdfs:subClassOf and rdf:type propagation. The noRDF engine runs semi-naive forward chaining to fixpoint, materializing every entailed triple with a justification chain.

Importantly, inference operates under the open-world assumption: the absence of a triple doesn't mean it's false, only unknown. This is the right model for regulatory knowledge, where data arrives incrementally and the ontology evolves.

2.2 Enforcement: What Can This Database Contain?

Cypher 25 introduced GRAPH TYPE—a schema declaration language that constrains what nodes, relationships, and properties the database will accept at write time. This is closed-world enforcement: if the schema says a Product must connect to an Ingredient via CONTAINS, writes that violate this constraint are rejected.

noRDF generates GRAPH TYPE DDL from the same ontology that drives projection. rdfs:subClassOf becomes label implications. rdfs:domain and rdfs:range become endpoint constraints. One ontology file—schema enforcement comes for free.

2.3 Computation: What Can Be Computed on Demand?

Some properties aren't stored or inferred—they're computed. The shortest path from a raw material to a finished product. The total concentration of an ingredient through a multi-level BOM. The number of hops to a regulatory category. These are graph-structural questions that Cypher answers in milliseconds.

noRDF captures these as computed properties in the ontology: Cypher recipes stored as RDF triples, evaluated on demand against the LPG projection.

§3. Architecture: Two Layers, One Database

The noRDF architecture is deliberately simple. Two layers coexist in a single Neo4j instance, connected only by URI keys. No cross-layer relationships. No mixed traversals.

3.1 The RDF Layer

Every RDF triple is stored as a (:Triple {s, p, o}) node with full URI strings. The subject, predicate, and object are plain string properties—no mapping, no abbreviation, no loss. Triples connect to (:Justification) nodes via [:PREMISE] and [:CONCLUDES] relationships, forming an AND-OR dependency graph.

This layer handles RDFS inference (forward chaining to fixpoint), justification-based truth maintenance (non-destructive retraction via DRed), and RDF I/O (Turtle, N-Triples, RDF/XML, JSON-LD via rdflib). It is a complete, spec-compliant RDFS engine that happens to run inside Neo4j.

3.2 The LPG Layer

Native Neo4j nodes and edges, projected from RDF triples by reading lpg: directives from the ontology. Each LPG node carries a uri property (indexed, unique) that serves as the join key back to the RDF layer. This layer is the target for Cypher queries, GDS algorithms, and GRAPH TYPE enforcement.

Crucially, the LPG layer does not store justifications, provenance, or class hierarchies. It stores only what the ontology says to project—the base facts that make structural computation efficient.

3.3 The Bridge: the lpg: Ontology

A minimal RDF vocabulary (~12 terms) at http://neo4j.com/lpg-ontology# that drives everything. It's stored as regular RDF triples—reasoned over by RDFS, interpreted by the engine.

One Ontology, Four Outputs

A single Turtle file produces four distinct artifacts:

  1. Projection — lpg:projectsAsLabel, lpg:projectsAsRelType, lpg:projectsProperty declare which RDF concepts become native LPG structures.
  2. Computation — lpg:ComputedProperty, lpg:ComputedRelationship with lpg:pattern define Cypher recipes evaluated on demand.
  3. Virtual Predicates — lpg:VirtualPredicate with lpg:transitiveClosureOf declares RDF properties backed by Cypher, with zero storage.
  4. GRAPH TYPE — GQL DDL generated from projection directives + RDFS class hierarchy. Write-time enforcement.

Here's what this looks like in practice: 

# Excerpt from the lpg: ontology — one file, four outputs

# 1. Projection: RDF class → Neo4j label
:Ingredient  lpg:projectsAsLabel  "Ingredient" .
:Product     lpg:projectsAsLabel  "Product" .

# 2. Projection: RDF property → Neo4j relationship
:contains    lpg:projectsAsRelType  "CONTAINS" .

# 3. Computed property: Cypher recipe, not stored triple
:bomDepth    a lpg:ComputedProperty ;
    lpg:pattern  "MATCH path = (x)-[:CONTAINS]->*(:RawMaterial)
                  RETURN max(length(path)) AS val" ;
    lpg:ownerVariable  "x" ;
    lpg:valueVariable  "val" .

# 4. Virtual predicate: transitive closure via Cypher
:hasComponent  a lpg:VirtualPredicate ;
    lpg:transitiveClosureOf  :contains .

§4. Design Principles

4.1 Don't Materialize the Chords of a Path

This is the rule that makes noRDF scale.

In graph theory, a chord is an edge connecting two non-adjacent vertices in a path. When you materialize a transitive closure (e.g., "Zeus hasAncestor Kronos" from a chain of childOf edges), every shortcut edge is a chord. For $n$ nodes in a linear chain, the closure produces $O(n^2)$ triples.

At 50,000 nodes with deep childOf trees, materializing hasAncestor as RDF triples produces millions of chords—each with a justification node, each consuming memory, each triggering further inference. The system grinds to a halt.

The solution is structural: project the base relationship (childOf → CHILD_OF) and declare a virtual predicate: 

:hasAncestor  a  lpg:VirtualPredicate ;
    lpg:transitiveClosureOf  :childOf .

Now hasAncestor resolves at query time via [:CHILD_OF]->* in Cypher—zero storage, sub-100ms at 50k nodes. The RDF predicate exists in the ontology. It has semantics. It can be queried through the same API. But it never produces a stored triple.

The principle: If a property can be computed from projected base facts via Cypher variable-length paths, don't materialize it. Project base facts. Compute derived ones.

4.2 Anchored Queries Scale Flat

A computed property anchored on a single entity runs in constant time, regardless of graph size. The Cypher engine follows edges from one starting node—the work is proportional to the local neighborhood, not the global graph.

The same computation for all entities is a full scan: $O(n)$ at best, catastrophic at worst. At 50,000 nodes, computing shortest paths for every entity was killed after 42 minutes.

The lpg:ownerVariable in the ontology tells the engine where to inject the URI filter. It's not a performance hint—it's a design requirement.

4.3 Sparse Is Better Than Dense

Not everything needs to be projected. If there's no lpg:projectsAsLabel, the class stays in the RDF layer only. If there's no lpg:projectsAsRelType, the relationship remains a reified triple. The ontology explicitly declares what becomes LPG structure—the rest stays as knowledge.

This is the opposite of an ETL mindset. You're not migrating data from one model to another. You're selectively surfacing structure where Cypher and GDS need it, while keeping the full RDF semantics available for reasoning.

4.4 Knowledge of Computation, Not Computed Knowledge

The RDF layer describes what a property means and how to compute it. The actual value lives in the LPG layer and is computed on demand. The ontology is a recipe book, not a result cache. 

# The ontology knows how to compute this...
:zeusNumber  a  lpg:ComputedProperty ;
    lpg:pattern  "OPTIONAL MATCH path = ANY SHORTEST (x)-[:CHILD_OF]->*
                  (z:Being {name: 'Zeus'})
                  RETURN length(path) AS val" ;
    lpg:ownerVariable  "x" ;
    lpg:valueVariable  "val" .

# ...but it never stores "Zeus zeusNumber 0" as a triple.

§5. Virtual Predicates: RDF Semantics, Cypher Performance

Virtual predicates are the key abstraction that lets the two layers collaborate. They're RDF properties with no stored triples—declared in the ontology, resolved transparently via Cypher on the LPG projection.

5.1 Transitive Closure (Concise)

The most common virtual predicate pattern. One triple in the ontology replaces millions of materialized chords: 

:hasAncestor  a  lpg:VirtualPredicate ;
    lpg:transitiveClosureOf  :childOf .

The engine reads transitiveClosureOf, looks up the projected relationship type for :childOf (CHILD_OF), and generates [:CHILD_OF]->+ for query resolution.

5.2 Pattern-Based (Flexible)

For virtual predicates that don't fit the transitive closure shorthand—inverse paths, multi-hop patterns, conditional traversals: 

:hasDescendant  a  lpg:VirtualPredicate ;
    lpg:pattern  "(s)<-[:CHILD_OF]-+(t)" ;
    lpg:sourceVariable  "s" ;
    lpg:targetVariable  "t" .

5.3 Transparent Query Interface

The caller doesn't know—or care—which layer answers. The query() method checks if a predicate is virtual and routes accordingly: 

# These two calls use the same API:
engine.query(s="myth:Zeus", p="rdf:type")         # → RDF layer (stored triples)
engine.query(s="myth:Zeus", p="myth:hasAncestor")  # → LPG layer (Cypher traversal)

This is the abstraction that makes the hybrid architecture invisible to consumers. Whether the answer comes from RDFS inference or Cypher traversal is an implementation detail.

§6. Demo: Cosmetics Supply Chain BOM

To show how this works in practice, consider a cosmetics formulation scenario. Products contain ingredients through a multi-level bill of materials (BOM). Each level has a concentration ratio. The final concentration of a raw ingredient in a finished product is the product of all ratios along the path.

6.1 The LPG Structure

The supply chain is a natural fit for LPG: products contain phases, phases contain pre-mixes, pre-mixes contain ingredients. Each CONTAINS relationship carries a ratio property. 

// Multi-level BOM with concentration ratios
MATCH path = (p:Product)-[rels:CONTAINS]->*(i:Ingredient)
WITH p, i, reduce(conc = 1.0, r IN rels | conc * r.ratio) AS finalConc
WHERE finalConc > 0
RETURN p.name AS product, i.name AS ingredient,
       round(finalConc * 100, 4) + '%' AS concentration

This is pure Cypher, pure LPG. Cypher's reduce() computes the cumulative ratio along variable-length paths—no RDF needed.

6.2 The RDF Knowledge

But ingredient classification is a different story. Retinol is a RetinoidAgent. RetinoidAgent is a subclass of ActiveIngredient. Retinoids are incompatible with AcidActiveAgents. These are logical relationships—class hierarchies, domain constraints, regulatory rules—that belong in RDF.

In the noRDF pattern, each ingredient carries its RDF knowledge as a Turtle property—a string containing valid RDF that the engine can project into an in-memory model for reasoning: 

// Ingredient node with embedded RDF knowledge
CREATE (retinol:Ingredient {
  name: 'Retinol',
  cas: '68-26-8',
  turtle: '
    @prefix cosmo: <http://example.org/cosmo#> .
    cosmo:Retinol rdf:type cosmo:RetinoidAgent ,
                           cosmo:PhotosensitiveAgent ;
                  cosmo:maxConcentrationEU "0.05" .
  '
})

6.3 Scope First, Reason Second

The key workflow: use Cypher to scope the data, then hand the relevant triples to the reasoning engine. For a regulation impact analysis: 

// 1. Cypher scopes: find all ingredients in EU-market products
MATCH (p:Product)-[:SOLD_IN]->(:Market {name: 'EU'})
MATCH path = (p)-[rels:CONTAINS]->*(i:Ingredient)
WHERE i.turtle IS NOT NULL

// 2. Project scoped triples into in-memory RDF graph
CALL n20s.graph.addTurtle('eu-check', i.turtle)

// 3. Add shared ontology (class hierarchy, rules)
MATCH (ont:Ontology)
CALL n20s.graph.addTurtle('eu-check', ont.turtle)

// 4. Query with backward-chaining RDFS inference
CALL n20s.graph.query('eu-check', '
  SELECT ?ingredient ?limit WHERE {
    ?ingredient a cosmo:RetinoidAgent ;
                cosmo:maxConcentrationEU ?limit .
  }', 'RDFS') YIELD row
RETURN row

Notice: Cypher narrows the problem to hundreds of triples (ingredients in EU products). The RDFS engine reasons over that subset—instant, even with full OWL semantics. This is the "scope first, reason second" pattern.

6.4 Regulation Blast Radius

When the EU restricts Retinol above 0.05%, the question isn't just "which products contain Retinol?" It's "which products contain any RetinoidAgent above the limit?" RDFS inference discovers that Retinal is also a RetinoidAgent (via subClassOf), even if no human flagged it explicitly.

The BOM traversal computes the final concentration. The RDFS reasoning identifies the regulatory class. Together, they answer a question neither paradigm could answer alone.

§7. n20s: GDS-Style In-Memory Reasoning

The supply chain demo above uses n20s—a Neo4j plugin that provides GDS-style in-memory RDF reasoning. The analogy is deliberate:

GDSn20s
gds.graph.project()n20s.graph.project()
gds.pageRank.stream()n20s.graph.query() (SPARQL)
gds.wcc.stream()n20s.graph.infer() (forward chaining)
—n20s.graph.query(..., 'RDFS') (backward chaining)
—n20s.graph.validate() (SHACL)
gds.graph.drop()n20s.graph.drop()

The mental model: Cypher selects and scopes triples from the LPG graph. n20s.graph.project() collects them into a named, heap-resident Apache Jena model. You query with SPARQL, run RDFS/OWL inference (forward or backward chaining), validate with SHACL, and drop the model when done. The in-memory graph is ephemeral—like a GDS projection.

7.1 Forward vs. Backward Chaining

n20s offers both inference strategies. The choice depends on the workflow:

Forward Chaining

Materialize all inferences first, then query the expanded graph. Best for multiple queries on the same projection.

// Step 1: materialize
CALL n20s.graph.infer(
  'scope', 'RDFS')

// Step 2+: query (fast)
CALL n20s.graph.query(
  'scope', 'SELECT ...')

Backward Chaining

Reason on-the-fly during query execution. Best for one-shot queries, simpler workflow.

// One step: reason + query
CALL n20s.graph.query(
  'scope',
  'SELECT ...',
  'RDFS')

7.2 SHACL Validation

SHACL shapes can be projected alongside data and ontology triples. The validate() procedure parses shapes from the graph and validates data against them, including SPARQL-based constraints: 

// Validate: SPARQL-based SHACL detects Retinoid + Acid co-formulation
CALL n20s.graph.validate('eu-check')
YIELD focusNode, severity, message
RETURN focusNode, severity, message

// → [Violation] Retinol — Retinoid incompatible with acid active agent
// → [Warning] AscorbicAcid — Missing EU max concentration limit

§8. Triples as Cargo, Not Structure

One of the more counterintuitive aspects of this architecture: RDF triples ride along as cargo on LPG nodes, invisible to normal Cypher queries. They don't pollute the graph with edges you'd never traverse. They don't create supernodes. They're just properties.

There are two cargo strategies, and the choice depends on your scoping needs:

Turtle Properties

Serialize RDF as a Turtle string property on the node. Clean, compact, no extra nodes.

(:Ingredient {
  name: 'Retinol',
  turtle: 'cosmo:Retinol
    a cosmo:RetinoidAgent .'
})

Best when: knowledge travels with the entity. Use n20s.graph.addTurtle() to project.

Triple Nodes

Store each triple as a (:Triple {s, p, o}) node connected via [:HAS_TRIPLE]. Granular, individually addressable.

(:Drug)-[:HAS_TRIPLE]->
(:Triple {
  s: 'drug:Aspirin',
  p: 'rdf:type',
  o: 'drug:NSAID'
})

Best when: you need fine-grained Cypher scoping of individual triples.

In both cases, the triples are invisible to standard graph traversals. A MATCH (p:Product)-[:CONTAINS]->(i:Ingredient) query sees the LPG structure. The RDF knowledge is only activated when you explicitly project it into an n20s in-memory graph.

§9. What We Learned

9.1 The Right Balance Is Selective Projection

The temptation is to project everything—if it exists as RDF, make it LPG. But projection has a cost: every projected relationship is a native edge with index overhead, lock contention, and visibility in graph algorithms. The right balance is to project only what Cypher and GDS need for structural computation, and keep everything else as RDF knowledge.

9.2 Justification Graphs Are Products

Unlike traditional triple stores that hide reasoning internals, the noRDF system makes the dependency graph a first-class object. "Why is Retinol classified as an ActiveIngredient?" is not a debugging question—it's a compliance question. The justification chain is the audit trail.

9.3 Two Paradigms, Never Mixed at Storage

The RDF layer and LPG layer share a database but never share structure. No relationships between :Triple nodes and :LPG nodes. The URI is the only join key. This prevents supernodes, cross-layer traversal pollution, and lock contention between reasoning and algorithmic workloads.

9.4 The Ontology Is the API Contract

Every computed property, every virtual predicate, every projection directive is declared in one Turtle file. The ontology is not documentation—it's executable. Change the ontology, and the projection changes, the computations change, the GRAPH TYPE changes. This is infrastructure as code for knowledge graphs.

§10. When to Use What

This is not a one-size-fits-all architecture. Here's a decision framework:

ScenarioApproachWhy
Static RDF data, Cypher queries onlyn10sMapping is fine when semantics are static
Live inference + graph algorithmsnoRDF (full engine)Need both paradigms with justification tracking
Scoped reasoning from LPG datan20s (in-memory)Cypher scopes, Jena reasons, results flow back
Ontology-driven LPG schemalpg: ontology → GRAPH TYPEOne artifact, write-time enforcement
Regulatory compliance, audit trailsnoRDF + justification graphThe "why" is the product

The honest answer is: most projects start with n10s, and that's the right call. noRDF becomes interesting when the ontology changes at runtime, when provenance is a requirement, when you need graph algorithms and inference on the same dataset, or when regulatory audit trails demand visible justification chains.

§11. Conclusion

The RDF-vs-LPG debate has always been a false dichotomy. RDF is not a data model you migrate away from. It's a reasoning paradigm that happens to store data as triples. LPG is not a replacement for RDF. It's a computational paradigm that happens to store data as nodes and edges.

noRDF is the observation that you don't have to choose. Keep both. Use RDF where logic works (inference, provenance, interoperability). Use LPG where structure matters (paths, centrality, algorithms). Bridge them with an ontology that drives both layers from a single file.

The name echoes NoSQL: not a rejection, but an expansion. Not only RDF. Not only LPG. Both, in one database, driven by one ontology.

noRDF: an RDF reasoning engine that knows when to stop being RDF. Semantic inference where logic works, graph algorithms where structure matters, schema enforcement where the database guarantees it—all in one Neo4j instance, driven by one ontology.
